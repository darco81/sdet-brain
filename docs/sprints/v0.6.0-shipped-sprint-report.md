# Sprint v0.6.0 - Image / PDF ingestion via DeepSeek-OCR

**Date:** 2026-05-14
**Tag:** `v0.6.0`
**Branch:** `feat/ocr-v0.6.0` → merged to `main`
**Repo:** PUBLIC at https://github.com/darco81/sdet-brain

## Goal

Add image and PDF ingestion to the brain. Receipts, invoices,
whiteboard photos, screenshots, and multi-page PDFs flow through the
same chunker → embedder → Qdrant path as markdown via a new OCR
subsystem with a layered provider chain.

The plan and edge-case matrix lives in
[`v0.6.0-ocr-feature-plan.md`](./v0.6.0-ocr-feature-plan.md).

## Architecture (1 core, 2 fronts)

```
src/sdet_brain/ocr/                    (NEW)
├── __init__.py                        public API exports
├── protocol.py                        IOCREngine + OCRResult + error hierarchy
├── factory.py                         get_ocr_engine - 3-tier fallback chain
├── mlx_vlm_provider.py                Apple Silicon DeepSeek-OCR-2-6bit
├── ollama_provider.py                 cross-platform HTTP fallback
└── prompts.py                         token-strip + dedup + quality helpers

src/sdet_brain/ingestion/
├── image_parser.py                    (NEW) parse_image + parse_pdf
└── pipeline.py                        (EXTEND) suffix-dispatched ingest

src/sdet_brain/server/tools/
└── ingest_image.py                    (NEW) explicit image/PDF MCP tool
```

## Fallback chain (Mac default)

1. **MLX-VLM + `mlx-community/DeepSeek-OCR-2-6bit`** — ~3 s/img on M5 Pro
2. **Ollama + `deepseek-ocr`** — same model, different backend, cross-platform
3. **Ollama + `qwen2.5-vl:32b`** — heavyweight multilingual (~40 s/img)

On health-check failure the factory walks down the chain. Win sister
release (next: `0.2.0-win.0`) sets `OCR_OLLAMA_FALLBACK_MODEL=` to
None and drops to a 1-link chain — Ollama DeepSeek only — because
4 GB VRAM cannot fit qwen2.5-vl:32b.

## Edge cases mitigated (plan §15)

1. EXIF rotation on phone photos → `PIL.ImageOps.exif_transpose`
2. HEIC support → `pillow_heif.register_heif_opener()` at module import
3. PDF >20 pages rejected → `OCR_MAX_PDF_PAGES` guard
4. Image >20 MB rejected → `OCR_MAX_IMAGE_BYTES` guard
5. Concurrent-ingest race → `threading.Lock` in factory singleton
6. Model-change idempotency → file-bytes `content_hash` + payload `ocr_model` mismatch
7. PII in receipts → feature flag `OCR_PII_SCRUB=false` (hook ready for v0.6.1)
8. OCR frontmatter auto-gen → `source=ocr`, `ocr_model`, `extracted_at`, `original_path`
9. Watcher picks up images accidentally → markdown-only `is_relevant_path` left intact
10. Memory pressure on Win → `OLLAMA_KEEP_ALIVE` default `5m`
11. Polish OCR quality → DeepSeek + Qwen2.5-VL both multilingual
12. Concurrent OCR contention → factory singleton (one engine per process)
13. Log flood → structured INFO per call with duration / model / peak
14. Search filter breakage → payload index on `source_type` covers `image-ocr`
15. MCP tool wrong-pick → explicit `ingest_image` docstring lists supported formats

## What's NOT in 0.6.0 (deferred)

- **Per-chunk `page_number`** in payload — PDF pages survive as
  `## Page N` markdown headings in chunker output; widening
  `Chunk` to track page index belongs in v0.6.1.
- **PII scrub** — feature flag wired, NER pipeline not yet built.
  Polish-specific entities (NIP, kwoty, dane osobowe) targeted for
  v0.6.1.
- **Live smoke test** with a real paragon — provider stack is
  end-to-end testable via mocks; real OCR validation belongs in
  manual QA on the user's M5 Pro before merging the branch.
- **CLI `--ocr-provider` flag** — REST + MCP auto-detect, CLI
  inherits via `ingest_path` but no per-call override yet.
